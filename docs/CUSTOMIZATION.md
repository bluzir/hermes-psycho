# Собери своего AI-психолога за 5 шагов

Это движок, а не готовый продукт под одну тему. Вертикаль «отношения» идёт как рабочий пример;
ниже — как переделать её под свой домен (тревога, самооценка, карьера, привычки, что угодно).

---

## Шаг 1. Форк и манифест профиля

`hermes-profile.yaml` — единственный источник правды. Из него `render-hermes-profile.py`
генерирует `hermes/SOUL.md`, `config.yaml.example` и bash-скрипты. Руками эти файлы не правят —
это считается drift и ловится `scripts/check-hermes-profile.sh`.

Что править в манифесте:

| Поле | Что задаёт |
|---|---|
| `owner.id` | идентификатор владельца (короткий slug) |
| `profile.principal` / `principal_dative` / `principal_genitive` | имя владельца в разных падежах — поставь `<ВАШЕ_ИМЯ>` и его формы |
| `profile.name` | имя профиля (напр. `relationship` → своё) |
| `profile.timezone` | таймзона владельца (влияет на крон-ритуалы) |
| `soul.role` | одно-строчная роль ассистента |
| `soul.domain` | домен («отношения» → «тревога» / «карьера» / …) |
| `soul.voice_address` | как агент обращается к владельцу |
| `soul.voice_tone` | тон персоны |
| `soul.voice_language` | язык (`ru` / …) |

После правок — перегенерируй артефакты:

```bash
python3 scripts/render-hermes-profile.py --write
```

## Шаг 2. Своя вертикаль вместо «отношений»

Слои `knowledge/` — общие психологические фреймворки. Большинство переносится на любой домен
без изменений.

**Универсальные — оставляй как есть:**

- `ifs/` — внутренние части, субличности в конфликте (работает на любой теме)
- `schema-therapy/` — ранние схемы, режимы, копинг-стили
- `self-regulation/` — самосаботаж, прокрастинация, страх отказа, Upper Limit
- `communication/` — Gottman / NVC, навыки диалога
- `systemic/` — роли, петли, границы, баланс agency

**Опциональные — зависят от домена:**

- `attachment/` — про близость и триггеры в контакте; для карьеры/привычек может быть не нужен
- `family/` — про семью происхождения; оставь, если домен затрагивает родителей/корни паттерна

**Добавить свой слой:**

1. Заведи каталог `knowledge/<твой-слой>/` с `00-index.md`.
2. Пиши страницы по формату `knowledge/schema.md` (frontmatter, `## Связи`, дедуп, эпистемические
   требования). Перед созданием — `grep` по теме, чтобы не плодить дубли (правила дедупа в схеме).
3. **Обнови `knowledge/RESOLVER.md`** — главный маршрутизатор. Добавь строки в таблицу
   «ситуация → оптика → протокол» и, если завёл новый слой, — в дерево решений «куда филить
   страницу» и в описание слоёв. Без этого агент не узнает про новую оптику.

## Шаг 3. Протоколы

`protocols/` — операционные плейбуки. Большинство домен-агностичны и переиспользуются как есть:

- `brain-first`, `multi-lens-read`, `co-thinking`, `memory`, `self-improvement`, `red-flags`,
  `crisis-support`, `weekly-review`, `life-interview` — работают на любой теме.
- `process-conversation`, `commitment-track`, `pre-conversation-memo`, `decision-support`,
  `add-context` — заточены под разбор отношений; под другой домен адаптируй формулировки триггеров
  и то, какие файлы они читают/пишут.

Реестр и цепочки — в `protocols/00-index.md`. Если появилась типовая ситуация без плейбука,
`new-request-playbook` описывает, как завести новый.

---

## Шаг 4. Тон и персона

Персона (`hermes/SOUL.md`) не редактируется напрямую — только через `hermes-profile.yaml`,
секция `soul.*` (роль, домен, обращение, тон, язык). После правки — пере-рендер:

```bash
python3 scripts/render-hermes-profile.py --write
```

Проверь результат в `hermes/SOUL.md` и, если нужно, догоняй формулировки в манифесте, а не в
сгенерированном файле.

---

## Шаг 5. Данные и деплой

1. Заведи локальные `people/` и `self/` из `_TEMPLATE`-шаблонов (эти каталоги в `.gitignore`).
   Посмотри выдуманный `examples/` как образец наполнения.
2. Разверни по ранбуку `deploy/relationship/README.md`: новый Telegram-бот, gbrain-индекс
   только по `knowledge/`, allowlist из одного владельца (single-user изоляция).
3. Прогони гейты:
   ```bash
   scripts/check-privacy.sh      # PII и секреты не под git
   scripts/check-structure.sh    # дерево репо соответствует раскладке
   ```

---

## Приватность

- Личные каталоги (`people/`, `self/`, `contexts/`, `data/`, аудио, транскрипты) — в `.gitignore`
  и **никогда** не эмбедятся: в семантический индекс идёт только `knowledge/`.
- `.privacy-denylist` — список реальных имён/slug'ов; `check-privacy.sh` роняет сборку, если любая
  из этих строк просочилась в git-tracked теорию. Держи его как guard от случайной вставки PII
  в страницу `knowledge/`.
- Прогоняй `check-privacy.sh` перед каждым коммитом.

## Тесты

После правок `hermes-profile.yaml` тесты рендера должны быть зелёными:

```bash
python3 -m unittest discover tests
```

Если рендер разошёлся с манифестом — тест это поймает. Правь source-of-truth
(`hermes-profile.yaml` + renderer), затем перегенерируй, затем прогоняй тесты.
