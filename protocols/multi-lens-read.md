# Протокол: мульти-оптика (multi-lens read)

**ТРИГГЕР:** Пользователь хочет разобраться в паттерне / ситуации глубже, чем «кто прав»:
«почему это снова происходит», «что за этим стоит», «посмотри с разных углов».
Также автоматически — при триггерах, ревности, тревоге в близости, повторяющихся
паттернах в любой связи.

**ВХОД:** описание ситуации / паттерна + slug человека (опционально файлы `map.md`, `conversations/`).
**ВЫХОД:** со-мышление в чате (≤ 15 строк на оптику + синтез); при необходимости предложение
обновить `map.md`. Никаких вердиктов — углы для пользователя.
**ОПТИКА:** RESOLVER → 2–3 слоя из `attachment/`, `ifs/`, `schema-therapy/`, `communication/`, `systemic/`.

---

## Шаги

### 1. Читать базу и контекст (brain-first — не пропускать)

- `knowledge/RESOLVER.md` → тип отношений + тип ситуации → конкретные каталоги.
- Прочитать 1–2 релевантных страницы `knowledge/<layer>/`.
- Если есть `people/<slug>/map.md` — прочитать раздел «Паттерны» и «Привязанность».
- **Если есть `self/map.md` — прочитать его перед разбором.** Там карта пользователя:
  стиль привязанности, доминирующие схемы, IFS-части, family-of-origin паттерны.
  Это основание для конкретного анализа вклада пользователя — не угадывать, а читать.
- Только после чтения — начинать разбор.

### 2. Выбрать 2–3 оптики (меню, не чек-лист)

Выбери те, что **клинически живы** для этой конкретной ситуации:

| Оптика | Когда брать |
|---|---|
| `attachment/` | триггер в контакте, тревога близости, дистанция, «стал холоднее» |
| `ifs/` | внутреннее противоречие, сильная реакция, «часть меня хочет / часть боится» |
| `schema-therapy/` | устойчивый паттерн, «это у меня везде», детский след |
| `communication/` | конфликт, недопонимание, паттерн диалога (критика / защита / уход) |
| `systemic/` | роли, асимметрия вклада, петли, «кто несёт ответственность» |

Три оптики — максимум; меньше и точнее лучше, чем шесть общих.

### 3. Разбор по каждой оптике

Для каждой оптики — отдельный блок. Структура блока:

```
**[Оптика: <название>]**
Что видно через этот угол:
• <наблюдение 1 с опорой на knowledge и на то, что сказал пользователь>
• <наблюдение 2>
Вклад пользователя в паттерн (обязательно): <конкретно и без осуждения>
```

**Правило вклада пользователя:** каждая оптика ОБЯЗАТЕЛЬНО включает, что пользователь привносит в
паттерн — не только поведение другого. Легко замечать чужое; его вклад называть прямо,
по делу, без морализаторства.

### 4. Синтез

После блоков оптик — синтез (≤ 5 строк):
- Что общего между оптиками: какой глубинный паттерн просвечивает.
- Один ключевой вопрос или угол, который пользователь, возможно, не видит.
- Что можно попробовать (одно действие / один разговор / одно наблюдение за собой).

Синтез — не вердикт. Пользователь решает что с этим делать.

### 5. Предложение по базе

- Если анализ выявил паттерн, которого нет в `map.md` — предложи обновить:
  «Добавить в map.md: паттерн X. Хочешь?»
- Если выявлена тема для `knowledge/` — предложи страницу (без PII).

---

## Анти-паттерны

- **Не проходи все оптики подряд формально** — только то, что реально освещает ситуацию.
- **Не обвиняй другого и не оправдывай пользователя.** «Оппонент по запросу» — режим, а не дефолт;
  но вклад пользователя называешь ВСЕГДА, мягко и прямо.
- **Не выдавай теорию, если не прочитал базу.** Пусто в knowledge — честно: «нет страницы,
  вот мои рассуждения».
- **Длинный разбор — в файл, не в чат.** Выжимка в чат, `<details>` или ссылку на файл.
