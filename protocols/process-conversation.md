# Протокол: обработка разговора

**ТРИГГЕР:** Пользователь прислал войс/текст разговора с кем-то («разбери разговор с X», «вот что было»),
либо Hermes автоматически транскрибировал голос — нужно извлечь, разобрать и записать.

**ВХОД:** транскрипт / текст разговора + имя или slug человека (опционально дата).
**ВЫХОД:** `people/<slug>/conversations/YYYY-MM-DD-тема.md`, обновление `map.md`, `commitments.md`, `next.md`.
**ОПТИКА:** RESOLVER → `communication/` + `systemic/` (разбор конфликта / разговора).

---

## Шаги

### 1. Определить человека и дату

- Slug из подписи («с Алексом», «по поводу Маши») — читаешь `people/_index.md` или
  `grep -rl "<имя>" /opt/.../people/` по полю `aliases:`.
- Slug не известен → один вопрос: «Кто это в базе? Нет ещё — дать имя?». При новом
  человеке сначала создать карточку (минимум `profile.md` + `map.md`), потом продолжить.
- Дата: из контекста или сегодня.
- **Мульти-участники** (разговор в группе, треугольник): определить всех slug'ов;
  одна `conversations/`-запись на каждого участника или один файл с cross-ссылкой —
  по смыслу.

### 2. Сохранить транскрипт

- Файл: `people/<slug>/conversations/YYYY-MM-DD-<тема-2-3-слова>.md`
- Шаблон:

```markdown
# <Дата> — <тема>
*slug: <slug> | тип: romantic / friend / cofounder*

## Дословно / краткий пересказ
<текст как есть или близкий пересказ>

## Ключевые моменты
- <момент 1>
- <момент 2>

## Что услышал / что сказал
<!-- паттерн коммуникации — по Gottman, NVC, systemic -->

## Коммитменты из разговора
<!-- обещания: кто / что / срок -->

## Открытые вопросы
<!-- что осталось неясным, что стоит прояснить -->
```

### 3. Читать базу перед разбором (brain-first)

- `knowledge/RESOLVER.md` → тип ситуации → оптика.
- Для разговора/конфликта: `knowledge/communication/` (Готтман, NVC, паттерны);
  `knowledge/systemic/` (роли, петли, баланс власти).
- Если в разговоре всплывают триггеры, тревога, дистанция → дополнительно
  `knowledge/attachment/`.

### 4. Разбор

Без флагов (см. `red-flags.md` — проверяешь всегда) — полный разбор:

- **Паттерн коммуникации:** назвать конкретно (критика, защита, contempt, stonewalling;
  bid/miss; repair attempt). Цитата из разговора → наблюдение.
- **Вклад пользователя в паттерн** — обязательный угол. Легко видеть другого; его часть
  называть прямо, без осуждения.
- **Системная рамка** (если уместно): роли, кто несёт ответственность за что,
  баланс agency.
- Выжимка пользователю в чат ≤ 10 строк: TL;DR + 1–2 ключевых наблюдения + следующий шаг.

### 5. Обновить `map.md`

- `people/<slug>/map.md` — динамика связи:
  - добавь/обнови раздел «Текущая динамика» (паттерн, стиль, что работает / что трёт).
  - если появились новые данные об attachment-стиле, схемах, ролях — обнови там же.
  - дата правки + короткое основание.

### 6. Обновить `commitments.md`

- Новые обещания из разговора (его и от другого) → `people/<slug>/commitments.md`.
- Формат таблицы — по `commitment-track.md`.
- Если нет — не трогаешь.

### 7. Обновить `next.md`

- `people/<slug>/next.md` — что сделать / сказать / прояснить до следующего контакта.
- Перезаписать раздел; предыдущее перенести в архивную строку, если выполнено.

### 8. Git + подтверждение

- Коммит не обязателен при каждом разборе; пользователь решает.
- Подтверждение одной строкой: «Разобрал разговор с <slug>, файлы обновлены».

---

## Правила

1. **Сначала сохранить, потом разбирать.** Файл пишется до любых выводов.
2. **Без PII в knowledge/.** Разбор — только в `people/` (gitignored). Паттерны без имён → в knowledge.
3. **Красные флаги — немедленно** (см. `red-flags.md`): ⚠️ отдельным сообщением пользователю
   до любой выжимки.
4. **Выжимка ≤ 10 строк.** Длинный анализ — в файл, ссылку дай пользователю.
5. **Мульти-связь:** если разговор касается нескольких людей — обнови карточки каждого.
