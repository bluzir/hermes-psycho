# Протокол: само-улучшение (сенсор пробелов)

Поверх обычной работы агент ПАССИВНО ловит, где система слаба, и копит данные для роста базы.
Пользователю вопросов не задаёт — это под капотом.

---

## Сигналы пробела (по убыванию надёжности)

1. **Явный фидбэк пользователя:** «не то / не полезно / не помогло / ответил не на то».
2. **Пользователь переспрашивает то же** за 2–3 хода (переформулирует ОДИН запрос, а не углубляет тему) — поведенческий сигнал «мне не хватило».
3. **Поиск по knowledge дал пусто** по нужной теме.
   Исключение: новый человек в базе — там тонко по определению (данных нет), это не сигнал.
4. **Ответ ушёл без чтения knowledge** (brain-first пропущен) — сильный сигнал. Сначала исправь (вернись и прочитай); если правда нечего — фиксируй.
5. **Самооценка «ответил из воздуха»** — слабый, вспомогательный.

---

## Что делать при сигнале (в тот же ход, тихо)

1. Дописать строку в `_gaps/queue.md`:
   `ДАТА · тема · чего не хватило · сигнал(1|2|3|4|5)`
   **Без реальных имён людей, без PII.** Только тема и что нужно.
2. Если сигнал фидбэка (1) — уточнить: какое измерение просело
   (механизм / паттерн / техника / системная рамка / коммуникационный ход).
3. Если это правка стиля или подхода пользователя → строка в `knowledge/_meta/preferences.md`
   (дата, суть, как применять дальше) — уже требование SOUL «Самообучение».

---

## Что НЕ делать

- Не задавать пользователю вопросов «чтобы закрыть пробел» — захват пассивный.
- Не писать реальные имена в `_gaps/queue.md`.
- Не канонизировать provisional-знание без явного согласия пользователя.
- Не отмечать нормальный «нет данных по новому человеку» как пробел системы.
