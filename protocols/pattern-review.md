# Протокол: обзор паттернов (pattern-review)

**ТРИГГЕР:** по запросу пользователя — «покажи паттерны», «что у меня повторяется»,
«какие у меня сквозные темы», «что видно по всей базе». On-demand, не по крону.

**ВХОД:** вся наполненная база пользователя — `self/map.md`, `self/spheres/*.md`,
`people/*/map.md`, `people/*/conversations/`. Читаешь то, что реально есть (grep + file).
**ВЫХОД:** 2–4 повторяющиеся темы ACROSS entities (не по одному человеку — сквозь несколько),
каждая как **гипотеза** с названным **вкладом пользователя**. Со-мышление в чате, не вердикт.
Если данных нет — честный empty-state (см. Шаг 0). Никаких выдуманных «опор» и хэшей.
**ОПТИКА:** RESOLVER → `attachment/` + `ifs/` + `schema-therapy/` + `systemic/` (логика `multi-lens-read`).

> **Это НЕ nightly `/patterns` из деплой-бота (gbrain).** Тот генерит дженерик-филлер по расписанию.
> Здесь — privacy-first разбор по реальной базе, по запросу, с честным «данных пока нет».

---

## Шаг 0 — Проверка данных (ДО любого разбора)

Прежде чем что-либо анализировать — убедись, что в базе есть реальные данные:

```
а) ls self/ people/ — есть ли что-то кроме _TEMPLATE / _index?
б) grep -rL "_TEMPLATE" self/map.md people/*/map.md — заполнены ли карты?
в) Есть ли хоть один people/<slug>/ с непустым map.md (не шаблон)?
```

**Если `self/` и `people/` держат только `_TEMPLATE` / нет реальных данных — СТОП.**
Не выдумывай паттерны. Ответь честно:

> В базе пока нет данных — паттерны выводить не из чего.
> Давай наполним: пройдём онбординг (`life-interview`) или добавим людей.
> Что ближе — рассказать о себе или добавить пару человек?

И НИЧЕГО фейкового: ни «опор», ни хэшей, ни изобретённых тем. Пустая база → пустой ответ + предложение.

**Если данных мало (1 человек, пара строк)** — скажи прямо: «сквозной паттерн пока не собрать,
данных на одну связь; давай добавим ещё или углубимся в неё через `multi-lens-read`».

## Шаги (когда данные есть)

### 1. Читать базу (brain-first — не пропускать)

- `knowledge/RESOLVER.md` → какие оптики живы для темы, которую пользователь принёс.
- **`self/map.md`** — карта пользователя: стиль привязанности, доминирующие схемы, IFS-части,
  family-of-origin. Это backbone: паттерн ищем не в вакууме, а относительно того, что уже знаем.
- `self/spheres/*.md` — паттерны по сферам (семья / романтика / работа / друзья).
- `people/*/map.md` — раздел «Паттерны» и «Привязанность» по каждому человеку.
- При необходимости — grep `people/*/conversations/` по повторяющимся словам/темам.

### 2. Найти повторы ACROSS entities (не within)

Ищешь не «паттерн Антона», а то, что всплывает **у 2+ разных людей / в 2+ сферах**:

- Одинаковый триггер в разных связях (например, «холодность» читается как отвержение везде).
- Одинаковый твой ход (например, уходишь в молчание при конфликте — и с мамой, и с партнёром).
- Одинаковая роль, которую занимаешь (миротворец / спасатель / тот, кто «слишком много даёт»).
- Один тип людей, которых выбираешь / притягиваешь.
- Расхождение поведение↔ценности, повторяющееся в разных контекстах.

Отбери 2–4 самых явных. Меньше и точнее лучше, чем список из восьми общих.

### 3. Оформить каждую тему как гипотезу

Для каждой темы — блок. Гипотеза, не диагноз. Структура:

```
**Тема N: <короткое имя паттерна>**
Где видно: <slug-1>, <slug-2> / сфера X, сфера Y (конкретные опоры из базы).
Через какую оптику: <attachment / ifs / schema / systemic> — что именно она подсвечивает.
Гипотеза: похоже на <паттерн> — <осторожная формулировка, «возможно», «выглядит как»>.
Твой вклад (обязательно): <что ты сам привносишь — конкретно, без осуждения>.
```

**Правило вклада (как в `multi-lens-read`):** каждая тема ОБЯЗАТЕЛЬНО называет, что привносит
сам пользователь — не только поведение других. Мягко и прямо, без морализаторства.

**Правило опор:** каждая тема опирается на реальные строки из `self/` / `people/`.
Нет опоры в базе → нет темы. Не додумывать, не обобщать до «у всех так».

### 4. Синтез (≤ 5 строк)

- Что общего между темами — какой один глубинный паттерн просвечивает через несколько.
- Один угол/вопрос, который пользователь, возможно, не видит про себя.
- Что можно попробовать: одно наблюдение за собой / один разговор. Не план из десяти пунктов.

Синтез — не вердикт. Пользователь решает, что с этим делать.

### 5. Предложения по базе / углублению

- Если выявлен сквозной паттерн, которого нет в `self/map.md` → предложи добавить:
  «Записать в self/map.md: паттерн X по нескольким связям. Хочешь?»
- Если пользователь хочет копнуть одну тему глубже → передай в `multi-lens-read`
  (одна ситуация, 2–3 оптики, полный разбор).

## Анти-паттерны

- **Пустая база → НЕ выдумывай.** Нет данных = честный empty-state (Шаг 0). Никаких фейковых
  «опор», хэшей, изобретённых тем. Это ключевое отличие от дженерик-`/patterns`.
- **Не паттерн по одному человеку.** Within-паттерн (один slug) — это `multi-lens-read`, не сюда.
  Здесь только сквозное, across 2+ сущностей.
- **Не диагноз.** Всё — гипотезы с пометкой. «Похоже на…», «возможно…».
- **Не обвиняй других, не оправдывай пользователя.** Вклад пользователя называешь ВСЕГДА.
- **Опора обязательна.** Каждая тема — из реальных строк базы. Нет строки → нет темы.
- **Длинный разбор — в файл.** В чат — выжимка; детали `<details>` или в `self/map.md`.

```

**Тема 1: <имя>**
Где: <slug>, <сфера> · Оптика: attachment
Гипотеза: ...
Твой вклад: ...

**Тема 2: <имя>**
...

**Синтез:** <один глубинный паттерн + один угол + один шаг>

Копнуть одну глубже (multi-lens) или записать в self/map.md?
```

**Данных нет:**

```
🔁 Паттерны

В базе пока нет данных — выводить не из чего.
Давай наполним: онбординг (расскажешь о себе) или добавим пару человек.
С чего начнём?
```
