# Протокол: brain-first

Базовое правило работы агента. Действует **перед любым разбором ситуации, паттерна
или созданием страницы knowledge**.
Цель: `knowledge/` — источник истины, не память чата.

---

## Триггер

Любой разбор:
- паттерна в связи (привязанность, схема, цикл);
- коммуникативного события (конфликт, разговор, дистанция);
- запроса «что за этим стоит» / «почему так происходит»;
- создания или обновления страницы в `knowledge/`.

---

## Шаги

### Перед разбором ситуации или паттерна

1. **Читать `knowledge/RESOLVER.md`** → определить тип ситуации → нужные каталоги и оптику.
2. **Прочитать 1–2 страницы** из указанных каталогов (`knowledge/<layer>/`).
   - `knowledge/` индексируется gbrain → `gb search "<тема>"` + `gb get <slug>`.
   - Fallback: `grep -ril "<тема>" /opt/.../knowledge/<dir>/` если MCP недоступен.
3. **Проверить карточку человека:** `people/<slug>/map.md` (паттерны, привязанность, история).
   `people/` — только grep/read, не через embedder.
4. **Опираться на прочитанное** — не на общие знания. Если страница расходится с интуицией
   чата — источник истины страница.
5. **Если страница не найдена** — честно: «страницы нет в knowledge, вот как рассуждаю».
   Не отвечать из воздуха.

### Перед созданием новой страницы knowledge

1. Читать `knowledge/RESOLVER.md` → определить каталог.
2. Дедуп: `grep -ril "<тема>" knowledge/<dir>/` (с алиасами).
3. Нашёл → UPDATE существующей страницы.
4. Не нашёл → CREATE по шаблону `knowledge/templates/page.md`.

---

## Фоллбэк

- **gbrain недоступен** → `grep -ril "<запрос>" knowledge/` по всем каталогам.
- **Страница не найдена** → сообщить пользователю, создать заготовку по шаблону.
- **RESOLVER не читается** → стандартное дерево: триггер в контакте → `attachment/`,
  паттерн повсюду → `schema-therapy/`, диалог → `communication/`, роли/петли → `systemic/`.
